# 소수몽키 독립 에이전트 도입계획

> 작성일: 2026-05-30
> 목적: Telegram API 기반 소수몽키 자료를 독립적으로 수집, 구조화, 판독하는 에이전트를 먼저 만든다.
> 원천 데이터: Telegram API 수집. 기존 `소수몽키/Telegram Desktop/ChatExport_2026-05-30`는 보조 검증 원천으로 유지한다.
> 현재 범위: 소수몽키 자료 자체의 이슈 브리핑, 종목/섹터 관찰, 개인 코멘트 성향을 독립적으로 정리한다.
> 확장 원칙: 처음부터 철학 체계나 10개 에이전트를 만들지 않는다. 반복 관점이 충분히 쌓이면 그때 분리한다.

PDF:

- `[신한]강세장 후반부 진입(소수의견).pdf`
- `[신한]기어세컨드 (소수의견).pdf`

## 1차 목표

```text
소수몽키 Telegram API
→ API 수집/인덱싱
→ 메시지/사진/PDF 연결
→ OCR 및 표 인식
→ 종목·ETF·이벤트·매크로 태그
→ 소수몽키 신호판
→ 소수몽키 독립 일일브리핑
```

1차 목표는 `소수몽키가 오늘 무엇을 중요하게 보고 있는가?`에 답하는 것이다.

## 초기 에이전트 구조

처음은 5개 에이전트로 시작한다.

| 번호 | 에이전트 | 역할 |
|------|----------|------|
| 00 | 소수몽키 총괄 | 전체 흐름, 호출 규칙, 산출물 위치를 관리한다. |
| 01 | 텔레그램 원천파서 | Telegram API에서 메시지, 사진, 파일을 읽고 날짜별 인덱스를 만든다. Desktop Export는 보조 검증에만 쓴다. |
| 02 | 이미지 OCR 판독 | 이미지 속 표, 히트맵, 티커, 수익률, 핵심 문장을 추출한다. |
| 03 | 신호 태그화 | 종목, ETF, 산업, 매크로, 이벤트, 실적 키워드를 붙이고 반복성을 본다. |
| 04 | 일일브리핑 | 소수몽키 관점에서 오늘 중요한 내용을 정리한다. |

소수몽키는 자산제곱처럼 명시된 철학 문서가 아니라 이슈 브리핑과 개인 코멘트 성향이 강하다. 따라서 철학 에이전트는 처음부터 만들지 않는다.

```text
이벤트:
강도:
근거 위치:

주의점:
다음 확인:
```

## 신호 등급

| 등급 | 의미 | 처리 |
|------|------|------|
| SM-S0 | 단순 소음 또는 반복성 낮음 | 기록 생략 또는 원자료 인덱스만 |
| SM-S1 | 배경 정보 | 일일 요약 후보 |

| SM-S3 | 소수몽키 관점에서 핵심 주장 또는 강한 변화 | 일일 보고 상단 노출 |

## 독립 판독 규칙

### 반복성

- 같은 종목, ETF, 산업, 매크로 축이 며칠 동안 반복되는가?
- 단발 뉴스인지, 시리즈로 이어지는 관점인지 구분한다.
- 반복 신호는 빈도와 최근성을 함께 기록한다.

### 강도

- 자료에서 강조된 정도를 본다.
- 표, 히트맵, PDF, 텍스트 설명이 같은 방향인지 본다.
- 단순 언급과 핵심 주장으로 쓰인 언급을 구분한다.

### 시장 반응

- 주가, ETF, 섹터 흐름이 이미 반응했는지 본다.
- 선행 아이디어인지, 이미 많이 오른 사후 해석인지 구분한다.
- 실적, 가이던스, 정책, 수급 중 어떤 촉매인지 분리한다.

### 일일 보고

```text
오늘의 소수몽키 핵심:
반복 등장 테마:
강한 종목/ETF:
새로 등장한 이벤트:
주의할 리스크:
다음 확인:
```

## 원천 데이터 처리 원칙

- 장기 저장 대상은 원문 전체가 아니라 태그, 요약, 신호, 근거 위치다.
- 이미지 OCR 결과는 원문 대체물이 아니라 분석 중간 산출물로 본다.
- 외부 비전 API 사용 전에는 별도 승인한다.
- 기본 PoC는 로컬 OCR부터 시작한다.

## API 수집 설계

기본 수집기는 Telethon을 사용한다.

| 필드 | 추출 기준 |
|------|-----------|
| 메시지 ID | Telegram message id |
| 일시 | Telegram message date를 `Asia/Seoul`로 변환 |
| 본문 | message text |
| 사진 | `--download-media`로 받은 원본 사진 |
| 파일 | `--download-media`로 받은 문서/첨부 |
| 서비스 메시지 | action message로 분리 집계 |

수집 결과는 `01_텔레그램_원천파서/API_수집/인덱스/` 아래에 둔다. OCR 단계에서는 `API_수집`을 원천 루트로 사용한다.

## HTML Export 보조 파서 설계

HTML Export는 JSON이 아니지만 구조가 일정하다.

| 필드 | 추출 위치 |
|------|-----------|
| 메시지 ID | `div.message`의 `id` |
| 일시 | `div.date.details`의 `title` |
| 본문 | `div.text` |
| 사진 | `a[href^="photos/"]` |
| PDF/파일 | `a[href^="files/"]` |
| 서비스 메시지 | `message service` |

메시지 묶음 규칙:

- 같은 분 단위로 연속된 사진은 하나의 카드 묶음으로 본다.
- 사진 직전/직후 텍스트는 해당 사진 묶음의 설명 후보로 연결한다.
- PDF는 별도 자료로 분리하되, Telegram 메시지의 게시 시각과 함께 보관한다.
- 썸네일은 분석 대상에서 제외하고 원본 이미지만 OCR한다.

## OCR 설계

1차는 로컬 `tesseract`를 사용한다.

```text
tesseract image.jpg stdout -l kor+eng --psm 6
```

예상 한계:

- 표 구조가 흐트러질 수 있다.
- 한글 문장 오타가 발생한다.
- 티커, 퍼센트, 종목명, 핵심 키워드는 비교적 추출 가능하다.

1차 PoC의 목표는 완전한 문장 복원이 아니라 아래 항목 추출이다.

- 티커
- 종목명
- ETF
- 상승률/하락률
- 실적/가이던스/CAPEX/금리/유가/정책 키워드
- 테마명

## 현재 저장 위치

```text
0_주식_에이전트/소수몽키_에이전트/
  소수몽키_에이전트_도입계획.md
  00_소수몽키_총괄/
  01_텔레그램_원천파서/
    API_수집/
  02_이미지_OCR_판독/
  03_신호_태그화/
  04_일일브리핑/

1_브리핑/소수몽키_브리핑/
  일일보고/
```

파서 인덱스, OCR 샘플, 신호판은 에이전트 작업 산출물로 `0_주식_에이전트/소수몽키_에이전트/` 내부에 둔다. 완성 일일브리핑만 `1_브리핑/소수몽키_브리핑/일일보고/`에 둔다.

API 세션과 `.env`는 원천파서 내부에 두되 공유하지 않는다. 원천 Export는 계속 아래 위치에 두고 API 장애 시 검증에만 쓴다.

```text
소수몽키/Telegram Desktop/ChatExport_YYYY-MM-DD
```

## PoC 단계

### 1단계: 구조 파악

- Telegram API 수집기 작성
- HTML 메시지 보조 파서 유지
- 날짜별 메시지 수 집계
- 사진 원본과 메시지 연결
- PDF 링크 연결

완료 기준:

- API 기준 요청 구간 메시지를 날짜별로 재구성한다.
- Export 기준 2026-04-01 ~ 2026-05-30 전체 메시지를 보조 검증한다.

### 2단계: OCR 샘플링

- 하루치 또는 최근 3일치 사진만 OCR
- 티커/종목/테마 추출률 확인
- 표 이미지와 히트맵 이미지를 분리

완료 기준:

- 이미지 자료에서 최소한 종목/ETF/테마/수익률을 추출할 수 있는지 판단한다.

### 3단계: 소수몽키 신호판

- SM-S0~SM-S3 등급화
- 날짜별 핵심 신호 저장
- 반복 언급 종목과 테마 빈도 집계

완료 기준:

- 하루치 소수몽키 신호판을 생성한다.

### 4단계: 소수몽키 일일 보고

- 소수몽키 관점의 일일 핵심 정리
- 반복 테마와 신규 테마 분리
- 강한 신호와 약한 신호 분리
- 다음 확인 포인트 기록

완료 기준:

- 2026-05-30 기준 소수몽키 독립 보고서를 만든다.

### 5단계: 검증과 반복 운용

- Telegram API 반복 운용
- 수동 Export는 장애 대응과 차이 검증에만 사용
- 계정 리스크, 약관, 저장 정책 재검토

완료 기준:

- API 수집이 안정적으로 같은 날짜 구간을 재현하는지 확인한다.

## 향후 확장 후보

현재는 만들지 않는다. 실제 자료를 며칠 이상 돌려본 뒤 필요할 때 추가한다.

| 후보 | 추가 조건 |
|------|-----------|
| 05 반복관점 추출 | 반복되는 선호 테마, 경계 리스크, 시장 해석 패턴이 보일 때 |
| 06 종목추적 | 특정 종목 언급 후 가격 반응과 후속 코멘트를 추적할 필요가 생길 때 |
| 07 PDF 리포트 판독 | PDF 자료 비중이 커지고 텔레그램 코멘트와 분리할 필요가 생길 때 |
| 08 검증규칙 | OCR 오류, 과잉 해석, 단발 이슈 오판이 반복될 때 |
| 09 외부 에이전트 연결 | 소수몽키 독립 에이전트가 안정적으로 일일보고를 만들 때 |

## 금지

- 소수몽키 신호만으로 매수/매도 결론을 내리지 않는다.
- 이미지 한 장의 분위기만으로 S3를 만들지 않는다.
- 아직 관찰되지 않은 철학이나 판단 체계를 임의로 만들지 않는다.
- 처음부터 10개 에이전트로 확장하지 않는다.
